# 계층 간 규칙

AnimaWorks에서 Anima 간 커뮤니케이션에는 조직 계층에 기반한 규칙이 있습니다.
이 문서에서는 각 관계(상사, 부하, 동료, 타 부서)에서의 규칙을 정의합니다.

## 기본 원칙

- 조직 내 모든 소통은 `send_message`(메시징)로 수행합니다
- 내부 상태의 직접 공유는 금지합니다. 자신의 말로 압축하여 해석해서 전달하세요
- 각 메시지에는 목적을 명확히 포함해야 합니다. "무엇을 해 주길 원하는지"와 "무엇을 전달하려는지"를 MUST 포함하세요

## 상사에 대한 커뮤니케이션 (보고, 연락, 상담)

상사는 `supervisor` 필드에 지정된 Anima입니다.
시스템 프롬프트의 "Supervisor" 섹션에 표시됩니다.

### MUST (의무)

- 태스크 완료 시 결과를 MUST 보고합니다
- 작업 중 문제나 차단 요인이 발생하면 즉시 MUST 보고합니다
- 자신의 판단 범위를 넘는 결정이 필요한 경우 MUST 상담합니다
- 상사의 지시가 불명확한 경우 MUST 확인합니다

### SHOULD (권장)

- 장기 태스크의 경우 중간 진행 상황을 SHOULD 보고합니다
- 문제 보고 시 "무엇이 발생했는지", "시도한 것", "제안"을 세트로 SHOULD 전달합니다
- 보고는 간결하게 합니다. 상세 내용이 필요한 경우 파일에 작성하여 참조를 SHOULD 보냅니다

### 메시지 예시

태스크 완료 보고:
```
To bob: API 엔드포인트 구현이 완료되었습니다.
- GET /api/users — 사용자 목록 조회
- POST /api/users — 새 사용자 생성
- 모든 테스트를 통과했습니다.
다음 태스크가 있으면 지시해 주세요.
```

문제 보고:
```
To bob: DB 연결에서 타임아웃이 빈번하게 발생하고 있습니다.
- 발생: 오늘 오후 2시경부터
- 시도한 것: 커넥션 풀 크기 확대 (개선 없음)
- 추정: 동시 접속 수가 상한에 도달한 가능성
- 제안: 커넥션 풀 최대값을 50에서 100으로 변경해도 될지 확인하고 싶습니다.
```

판단 상담:
```
To alice: 캐싱 전략에 대한 판단을 부탁드립니다.
- 옵션 A: Redis 도입 (빠르지만 인프라 비용 증가)
- 옵션 B: 파일 캐시 (저비용이지만 속도에 한계)
- 저는 A를 추천하지만, 비용 관련 판단은 맡기겠습니다.
```

## supervisor 도구

부하를 가진 Anima에게는 조직 관리를 위한 전용 도구가 자동으로 활성화됩니다.

### 도구 목록

| 도구 | 대상 범위 | 개요 | 주요 파라미터 |
|------|----------|------|---------------|
| `org_dashboard` | 전체 배하(재귀) | 배하 전체의 프로세스 상태, 최종 활동, 현재 태스크, 태스크 수를 트리로 표시 | 없음 |
| `ping_subordinate` | 전체 배하(재귀) | 배하의 생존 확인. `name` 생략 시 전원 일괄, 지정 시 단일 Anima | `name` (선택) |
| `read_subordinate_state` | 전체 배하(재귀) | 배하의 `state/current_state.md` 읽기 | `name` (필수) |
| `delegate_task` | 전체 배하(재귀) | 태스크 위임 (배하 큐 추가 + DM 전송 + 자신 측 추적 엔트리 생성) | `name`, `instruction`, `deadline` (필수), `summary` (선택) |
| `task_tracker` | 자신의 위임 태스크 | `delegate_task`로 위임한 태스크의 진행 상황을 배하 측 큐에서 추적 | `status` (선택: "all"/"active"/"completed", 기본값 "active") |
| `audit_subordinate` | 전체 배하(재귀) | 활동 타임라인 또는 통계 요약을 생성. `name` 생략 시 전체 배하를 일괄 감사 (통합 타임라인) | `name` (선택), `mode` (선택: `"report"`/`"summary"`, 기본값 `"report"`), `hours` (선택: 1~168, 기본값 24), `direct_only` (선택: boolean), `since` (선택: `"HH:MM"` 오늘의 시작 시각, 지정 시 hours보다 우선) |
| `disable_subordinate` | 전체 배하(재귀) | 배하를 정지 (status.json enabled=false, 약 30초 후 프로세스 정지) | `name` (필수), `reason` (선택) |
| `enable_subordinate` | 전체 배하(재귀) | 정지된 배하를 재개 | `name` (필수) |
| `set_subordinate_model` | 전체 배하(재귀) | 배하의 모델을 변경 (status.json 업데이트. 반영에는 `restart_subordinate`가 필요) | `name`, `model` (필수), `reason` (선택) |
| `set_subordinate_background_model` | 전체 배하(재귀) | 배하의 백그라운드 모델(Heartbeat/Inbox/Cron용)을 변경. 빈 문자열로 초기화. 반영에는 `restart_subordinate`가 필요 | `name`, `model` (필수), `credential`, `reason` (선택) |
| `restart_subordinate` | 전체 배하(재귀) | 배하 프로세스를 재시작 (restart_requested 플래그, 약 30초 후 재시작) | `name` (필수), `reason` (선택) |

`check_permissions`는 모든 Anima가 사용 가능합니다 (자신의 권한 목록을 확인).

### 태스크 위임 플로우

1. `delegate_task(name="dave", instruction="...", deadline="...")`를 실행
   - `deadline`은 상대 형식(`30m`, `2h`, `1d`) 또는 ISO8601 형식(예: `2026-02-20`)
   - `summary`는 선택 (생략 시 instruction의 처음 100자)
2. dave의 태스크 큐(`state/task_queue.jsonl`)에 태스크가 자동 추가됩니다
3. dave의 `state/pending/`에 태스크 JSON이 기록되어 즉시 실행됩니다
4. dave에게 DM이 자동 전송됩니다
5. 자신의 큐에 추적 엔트리가 생성됩니다 (status="delegated")
6. `task_tracker(status="active")`로 진행 중인 위임 태스크를 추적합니다 (`status="all"`로 전체, `status="completed"`로 완료분만)

### 배하 상태 확인 (정기적으로 수행해야 함)

```
org_dashboard()                        # 배하 전체 상태를 트리로 표시
read_subordinate_state(name="dave")    # dave의 현재 태스크와 상태 확인
ping_subordinate()                     # 전원의 생존 확인
audit_subordinate(name="dave")         # dave의 최근 24시간 활동 타임라인
audit_subordinate(name="dave", mode="summary", hours=168)  # dave의 최근 7일간 통계 요약
audit_subordinate()                    # 전체 배하의 통합 타임라인 (최근 24시간)
audit_subordinate(direct_only=true)    # 직속 부하만의 통합 타임라인
audit_subordinate(since="09:00")       # 전체 배하의 오늘 9시 이후
audit_subordinate(name="dave", since="13:00")  # dave의 오늘 13시 이후
```

#### audit_subordinate 모드

| 모드 | 출력 내용 | 용도 |
|------|----------|------|
| `report` (기본값) | 시계열 타임라인. 이벤트별 아이콘, 시각, 내용 표시. 도구 사용은 하단에 집약 요약 | "오늘 무엇을 했는지" 파악 |
| `summary` | 이벤트 수, 태스크 상황, 통신 상대, 에러 상세의 통계 표시 | 정량적 활동 평가 |

`name`을 생략하여 여러 Anima가 대상이 되는 경우, `report` 모드에서는 모든 Anima의 이벤트를 시계열로 병합한 **통합 타임라인**이 생성됩니다.

### 조직 확장 (신규 Anima 생성)

`skills/newstaff/SKILL.md`를 보유한 Anima는 `create_anima` 도구로 신규 Anima를 생성할 수 있습니다.
캐릭터 시트(`character_sheet_content` 또는 `character_sheet_path`)를 지정하세요.
생성 후 config.json에 자동 등록되며 서버 리로드 시 활성화됩니다.
`supervisor`를 생략하면 호출한 Anima가 상사로 설정됩니다.

## 부하에 대한 커뮤니케이션 (지시, 확인)

부하는 시스템 프롬프트의 "Subordinates" 섹션에 표시되는 Anima입니다.
자신을 `supervisor`로 설정한 모든 Anima가 해당합니다.

### MUST (의무)

- 태스크 지시 시 다음을 MUST 포함합니다:
  - **목적**: 왜 이 태스크가 필요한지
  - **기대하는 산출물**: 무엇을 만들거나 할 것인지
  - **산출물 경로**: 위임 대상의 디렉토리 또는 `common_knowledge/` (자신의 `knowledge/`는 지정 불가 — 부하는 기록할 수 없음)
  - **기한 및 우선순위**: 언제까지, 얼마나 긴급한지
- 부하의 완료 보고를 받으면 확인 결과를 MUST 피드백합니다
- 정기적으로 `org_dashboard`로 배하 상태를 MUST 확인합니다

### SHOULD (권장)

- 부하의 speciality를 고려하여 태스크를 SHOULD 할당합니다
- 지시는 구체적으로 합니다. 모호한 표현("적당히 해 줘")은 SHOULD 피합니다
- 부하가 여러 명인 경우, 태스크의 의존 관계를 SHOULD 정리한 후 지시합니다

### MAY (선택)

- 부하의 성장을 위해 약간 도전적인 태스크를 MAY 할당합니다
- 부하 간 연계가 필요한 경우, 관계자를 명시하여 MAY 지시합니다

### 메시지 예시

태스크 지시:
```
To dave: 사용자 인증 API 구현을 부탁합니다.
- 목적: 프론트엔드에서의 로그인 기능을 구현하기 위해
- 산출물: POST /api/auth/login 엔드포인트 (JWT 반환)
- 사양: 이메일 + 비밀번호 인증. 실패 시 401 반환
- 기한: 이번 주 금요일까지
- 관련: eve가 프론트엔드 쪽 로그인 화면을 병렬로 작성 중입니다.
  API 사양에 대한 질문이 있으면 eve와 직접 소통해도 됩니다.
```

확인 및 피드백:
```
To dave: 인증 API 구현을 확인했습니다.
- 로그인/로그아웃 모두 정상 동작 확인
- 1가지: 토큰 만료 기간이 설정되지 않은 것 같습니다. 24시간으로 만료 기간을 설정해 주세요
- 그 외에는 문제 없습니다. 수정 후 다시 보고해 주세요.
```

## 동료와의 커뮤니케이션 (연계, 조율)

동료는 같은 `supervisor`를 가진 Anima입니다.
시스템 프롬프트의 "Peers" 섹션에 표시됩니다.

### SHOULD (권장)

- 관련 업무를 담당하는 동료와는 직접 소통을 SHOULD 수행합니다
- 자신의 작업이 동료의 작업에 영향을 미치는 경우 사전에 SHOULD 공유합니다
- 공동 작업 시 누가 무엇을 담당하는지 SHOULD 명확히 합니다

### MAY (선택)

- 동료의 speciality에 관한 지식을 MAY 상담합니다
- 리뷰나 의견을 MAY 요청합니다

### 메시지 예시

연계 요청:
```
To eve: 인증 API 구현이 완료되어 공유합니다.
- 엔드포인트: POST /api/auth/login
- 요청: { "email": "...", "password": "..." }
- 응답: { "token": "JWT 문자열", "expires_in": 86400 }
- 에러 시: 401 { "error": "Invalid credentials" }
프론트엔드 쪽 로그인 화면에서 사용해 주세요. 질문이 있으면 언제든 연락하세요.
```

상담:
```
To carol: 관리 화면의 UI에 대해 상담하고 싶습니다.
사용자 목록 화면의 테이블 레이아웃에 대해,
20개 이상의 컬럼이 있는데 어떻게 표시하는 것이 좋을까요?
UX 관점에서 조언을 부탁드립니다.
```

## 타 부서 멤버에 대한 커뮤니케이션

타 부서란 자신과 다른 `supervisor`를 가진 Anima를 의미합니다.
동료 섹션에 표시되지 않는 Anima가 해당합니다.

### MUST (의무)

- 타 부서 멤버에게 직접 연락하는 것은 원칙적으로 MUST 하지 않습니다
- 타 부서와의 연계가 필요한 경우, 자신의 상사에게 그 사실을 MUST 전달합니다
- 상사를 통해 상대 부서의 상사에게 연락이 가고, 거기서 상대에게 지시가 내려옵니다

### 커뮤니케이션 경로

예: dave(bob 배하)가 frank(carol 배하)에게 연락하고 싶은 경우

```
올바른 경로:
  dave → bob(자신의 상사) → alice(공통 상사) → carol(frank의 상사) → frank

잘못된 경로:
  dave → frank (직접 연락은 금지)
```

### 메시지 예시

상사에 대한 요청 (타 부서 연계가 필요한 경우):
```
To bob: 고객 데이터 내보내기 기능에 대해,
영업부의 frank 씨가 필요로 하는 CSV 형식의 상세 내용을 알고 싶습니다.
carol 씨를 통해 확인해 주실 수 있나요?
```

### 직접 연락을 피해야 하는 이유

- 상사가 업무 전체 그림을 파악할 수 없게 됩니다
- 지시 체계가 혼란스러워져 상충되는 지시가 나올 위험이 있습니다
- 각 부서의 상사가 우선순위를 판단할 기회가 상실됩니다

## 최상위 Anima의 사람에 대한 연락

상사(supervisor)가 설정되지 않은 최상위 Anima는 사람 관리자에게 직접 연락할 책임이 있습니다.
연락에는 `call_human` 도구를 사용합니다.

### MUST (의무)

- 문제, 에러, 장애를 감지한 경우 `call_human`으로 MUST 연락합니다
- 자신의 판단 범위를 넘는 결정이 필요한 경우 MUST 보고합니다
- 부하로부터 에스컬레이션된 사항을 스스로 해결할 수 없는 경우 MUST 보고합니다

### SHOULD (권장)

- 중요한 태스크의 완료 시 SHOULD 보고합니다
- 부하의 업무 상황에 우려가 있는 경우 SHOULD 보고합니다

### MAY (선택)

- 정상적인 순찰에서 특별한 문제가 없었던 경우 보고를 MAY 생략합니다
- 경미한 자동 복구가 완료된 경우 보고를 MAY 생략합니다

### 메시지 예시

문제 보고:
```
call_human:
  subject: "프로덕션 환경 API 응답 지연 감지"
  body: |
    오후 2시경부터 API 응답 시간이 평소의 3배로 증가했습니다.
    - 영향: 사용자 대상 API의 모든 엔드포인트
    - 시도한 것: 커넥션 풀 상태 확인 (이상 없음)
    - 추정: DB 서버 부하가 높을 가능성
    - 제안: DB 서버 스케일업을 검토해 주세요
  priority: high
```

판단 요청:
```
call_human:
  subject: "신규 기능 릴리스 승인 요청"
  body: |
    사용자 인증 기능 구현이 완료되어 모든 테스트를 통과했습니다.
    프로덕션 환경 릴리스에 대한 승인을 부탁드립니다.
  priority: normal
```

## 긴급 시 예외 규칙

일반적인 계층 규칙에는 다음과 같은 예외가 있습니다.

### 긴급 상황의 정의

다음에 해당하는 경우 "긴급 상황"으로서 예외 규칙을 MAY 적용합니다:

- 시스템 장애나 서비스 중단이 발생한 경우
- 보안 인시던트가 발생한 경우
- 상사가 장시간 응답하지 않는 상태가 지속되는 경우
- 기한이 임박하여 정상 경로로는 간에 맞지 않는 경우

### 긴급 시 허용되는 행동

- 타 부서 멤버에게 직접 연락 MAY (단, 사후에 상사에게 MUST 보고)
- 상사를 건너뛰어 그 위의 상사에게 직접 연락 MAY (단, 직속 상사에게도 MUST 보고)
- 통상적으로 상담이 필요한 판단을 스스로 내리는 것을 MAY 허용 (단, 사후에 MUST 보고 및 승인 획득)

### 긴급 연락 메시지 예시

```
To frank: [긴급] dave입니다. 통상은 상사 경유이지만, 긴급 상황이라 직접 연락드립니다.
프로덕션 환경에서 사용자 데이터 불일치가 발생했습니다.
영업 쪽에서 고객으로부터 문의가 오지 않았는지 확인해 주실 수 있나요?
bob과 carol에게도 동시에 보고하고 있습니다.
```

상사에 대한 사후 보고:
```
To bob: [사후 보고] 프로덕션 장애 대응에서 frank 씨(carol 배하)에게 직접 연락했습니다.
- 사유: 고객 데이터 불일치의 영향 범위를 긴급 확인할 필요가 있었습니다
- 대응 내용: frank 씨에게 고객 문의 상황 확인을 요청
- 결과: 3건의 문의가 와 있으며, frank 씨가 대응 중
사후 보고가 되어 죄송합니다. 확인 부탁드립니다.
```

## 규칙 우선순위 정리

| 우선도 | 규칙 | 분류 |
|--------|------|------|
| 1 | 상사에 대한 문제 보고 및 에스컬레이션 | MUST |
| 2 | 태스크 완료 보고 | MUST |
| 3 | 부하에 대한 지시에 목적, 산출물, 기한 포함 | MUST |
| 4 | 타 부서에 대한 직접 연락 금지 | MUST |
| 5 | 동료와의 직접 연계 | SHOULD |
| 6 | 중간 진행 상황 보고 | SHOULD |
| 7 | 긴급 시 예외 적용 | MAY |
