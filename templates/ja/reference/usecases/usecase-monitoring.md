# ユースケース: インフラ・サービス監視

サーバー、Webサービス、クラウドリソースなどを24時間365日監視し、異常検知・アラート・初動対応を自動化するユースケースです。

---

## こんな課題を解決できる

- サーバーダウンに気づくのが遅い
- 夜間・休日の障害対応ができない
- SSL証明書の期限切れを見落とす
- ディスク容量やリソースの枯渇に気づかない
- 監視ツールのアラートが多すぎて重要なものを見逃す

---

## パターン1: Webサービスの死活監視

### やること
Webサービスが正常に応答しているかを定期的にチェックし、異常時にアラートを発報します。

### 動作の流れ
1. 定期間隔（例: 5〜30分ごと）で対象URLにアクセス
2. HTTP応答コード・応答時間を確認
3. 異常を検知した場合：
   - 1回目: 再確認（一時的なエラーの可能性）
   - 2回連続: 人間に通知
   - 復旧を検知したら復旧通知を送信
4. 応答時間の推移を記録し、劣化傾向を検知

### 活用例
- 本番Webサイトが503を返した → 即座に管理者のスマホに通知
- APIの応答が通常の3倍遅くなった → パフォーマンス劣化の警告
- メンテナンス終了後の復旧確認 → 正常復帰を自動確認・報告

### 始めるのに必要なもの
- 監視対象のURL一覧
- アラート通知先（チャット、メール等）
- 判定基準（何秒以上で遅延とみなすか、等）

---

## パターン2: サーバーリソース監視

### やること
CPU、メモリ、ディスク容量、プロセス状態などのサーバーリソースを監視します。

### 動作の流れ
1. 定期的にサーバーの状態を取得（SSH or API経由）
2. 各指標を閾値と比較：
   - ディスク使用率 > 80% → 警告
   - ディスク使用率 > 95% → 緊急通知
   - メモリ使用率の急激な上昇 → メモリリークの疑い
   - CPU高負荷の持続 → プロセスの暴走チェック
3. 異常があればアラートと推奨対応を通知

### 活用例
- ディスク容量が残り5%になった → 古いログの削除候補を提示
- 特定プロセスがCPU 100%で張り付いている → プロセス名と起動時間を報告
- メモリ使用量が日々増加している → リークの可能性を警告

### 自動対応の例
- ログローテーション（古いログファイルの圧縮・削除）
- 暴走プロセスの特定と報告（自動キルは人間の承認後）
- 一時ファイルのクリーンアップ

---

## パターン3: SSL/TLS証明書の期限監視

### やること
SSL証明書の有効期限を定期チェックし、期限切れ前に更新を促します。

### 動作の流れ
1. 毎日1回、対象ドメインの証明書有効期限を確認
2. 残り日数に応じてアラートレベルを変更：
   - 残り30日 → 情報通知（更新準備を推奨）
   - 残り14日 → 警告通知（早急に更新を）
   - 残り7日 → 緊急通知（即座に対応が必要）
3. 更新手順を記載したリマインダーを送信

### 活用例
- 複数ドメインの証明書期限を一括管理
- Let's Encrypt等の自動更新が正常に動いているかの確認
- 更新後の証明書が正しく適用されているかの検証

---

## パターン4: クラウドサービスの監視

### やること
クラウド上のリソース（コンテナ、データベース、ストレージ等）の状態と費用を監視します。

### 動作の流れ
1. クラウドAPIを使ってリソース状態を定期取得
2. 以下を監視：
   - サービスの稼働状態（正常/異常）
   - エラーログの発生状況
   - コスト推移（予算超過の検知）
3. 異常や予算超過を検知したら通知

### 活用例
- コンテナサービスのタスク数が減少 → サービス障害の疑い
- データベースのコネクション数が上限に近い → スケールアップの提案
- 今月のクラウド費用が予算の80%を超えた → コスト警告

---

## パターン5: ログ分析と異常検知

### やること
アプリケーションログを定期的に分析し、エラーパターンや異常な傾向を検知します。

### 動作の流れ
1. 指定期間のログを取得
2. エラー・警告の発生件数をカウント
3. 通常時との比較で異常を判定：
   - エラー率が通常の3倍 → 調査アラート
   - 新しいタイプのエラーが発生 → 新規障害の疑い
   - 特定のエンドポイントでエラー集中 → 該当機能の障害
4. 分析結果をサマリーとして報告

### 活用例
- 「過去30分でエラー0件、正常」→ 定常報告
- 「新しいエラーパターンを3件検知」→ 詳細を添えて報告
- 「特定APIのエラー率が急上昇」→ 原因調査を開始

---

## パターン6: 統合ダッシュボード的な運用

### やること
複数の監視結果を統合し、定期的にサマリーレポートを生成します。

### 動作の流れ
1. 各監視担当からの報告を集約
2. 全体のステータスを一目で分かるサマリーにまとめる
3. 定期（例: 毎朝9時）に人間に報告

### レポート例
```
== 本日のインフラ状況 ==
[正常] Webサービス: 応答正常（平均120ms）
[正常] サーバーリソース: CPU 15%, メモリ 45%, ディスク 62%
[注意] SSL証明書: example.com の期限まで残り20日
[正常] クラウドサービス: 全タスク稼働中
[正常] エラーログ: 過去24時間のエラー 0件
```

## 構成のヒント

### 最小構成（Anima 1体）
- 1体で全項目を巡回監視
- 巡回間隔は30分程度
- 小規模環境（サーバー1〜2台）なら十分

### 推奨構成（Anima 3〜4体）
- **監視統括**: 全体の状態把握、異常判断、エスカレーション
- **サーバー監視**: OS、プロセス、リソースの監視
- **ネットワーク/SSL監視**: SSL証明書、DNS、接続性の監視
- **クラウド監視**: クラウドサービス固有の監視

### 誤報を減らすコツ
- 1回の異常で即アラートにしない（2回連続で確認）
- 閾値は実環境の平常値を基準に設定する
- 監視Animaの報告を統括が独立検証する体制にする
- 誤報パターンを知識として蓄積し、判定精度を向上させる
